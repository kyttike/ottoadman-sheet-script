# Clasp Node Project Design

**Date:** 2026-03-20
**Status:** Approved

## Overview

Convert the existing Google Apps Script project (`Import.gs`) to a TypeScript-first local development workflow. TypeScript source is bundled via esbuild into a single `dist/Code.js` file, which is pushed to Google Apps Script using clasp.

## Goals

- Write and maintain the script in TypeScript locally
- Bundle TypeScript into a single `.gs`-compatible file using esbuild
- Push to Google Apps Script via clasp from the `dist/` directory
- Support future automated deployments via CLI/CI

## Project Structure

```
ottoadman-sheet-script/
├── src/
│   ├── index.ts        # Entry point — assigns GAS functions to globalThis
│   ├── api.ts          # BitJita API fetch helpers
│   ├── sheet.ts        # Spreadsheet read/write helpers
│   └── update.ts       # Core update() / setup() / populateConfig() logic
├── dist/               # esbuild output — only Code.js is gitignored
│   ├── Code.js         # Generated by esbuild — gitignored
│   └── appsscript.json # GAS manifest — committed to repo
├── .clasp.json         # rootDir: "dist", scriptId
├── build.js            # esbuild build script (Node CJS)
├── package.json
├── tsconfig.json
└── .gitignore
```

### `.gitignore` changes required

The existing `.gitignore` has a bare `dist` entry (line 83, under the `# Nuxt.js build / generate output` comment) that excludes the entire `dist/` directory. Replace that line with:

```
dist/Code.js
```

This ensures `dist/appsscript.json` is committed while the generated bundle is not.

Setup order matters:

1. Create `dist/` directory and write `dist/appsscript.json` (see contents below)
2. Update `.gitignore` to replace the bare `dist` line with `dist/Code.js`
3. Force-add the manifest since the directory was previously excluded:

```sh
git add -f dist/appsscript.json
```

## Build Pipeline

- **Bundler:** esbuild (JS API via `build.js`)
- **Entry:** `src/index.ts`
- **Output:** `dist/Code.js`
- **Format:** `iife` — required for GAS (no ES module support)
- **Platform:** `browser` (GAS runs in a browser-like V8 environment)
- **Target:** `es2019` (compatible with GAS V8 runtime)
- **Bundle:** `true` — inlines all imports into a single file
- **Tree shaking:** disabled — ensures `globalThis` assignments in `index.ts` are not eliminated

## npm Scripts

| Script | Command | Purpose |
|--------|---------|---------|
| `build` | `node build.js` | Compile TypeScript → `dist/Code.js` |
| `typecheck` | `tsc --noEmit` | Type-check source without emitting files |
| `push` | `clasp push` | Push `dist/` to Google Apps Script |
| `deploy` | `npm run build && npm run push` | Build and push in one step |


## TypeScript Configuration (`tsconfig.json`)

```json
{
  "compilerOptions": {
    "target": "ES2019",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "noEmit": true,
    "lib": ["ES2019"],
    "types": ["google-apps-script"]
  },
  "include": ["src/**/*.ts"],
  "exclude": ["node_modules", "dist"]
}
```

Key points:
- `noEmit: true` — TypeScript is used for type-checking only; esbuild handles transpilation
- `module: "ESNext"` + `moduleResolution: "bundler"` — lets esbuild resolve ES module imports
- `lib: ["ES2019"]` — matches the GAS V8 runtime; avoids pulling in DOM types
- `types: ["google-apps-script"]` — enables GAS API type definitions

## `build.js`

CommonJS Node script using the esbuild JS API:

```javascript
const esbuild = require('esbuild');

esbuild.build({
  entryPoints: ['src/index.ts'],
  outfile: 'dist/Code.js',
  bundle: true,
  format: 'iife',
  platform: 'browser',
  target: 'es2019',
  treeShaking: false,
}).catch(() => process.exit(1));
```

`treeShaking: false` ensures the `globalThis.setup = setup` (and similar) assignments in `index.ts` are not eliminated. Note: esbuild's iife format generally treats side-effecting statements as safe, so disabling tree shaking is a conservative choice — the known trade-off is that unused code will not be eliminated.

## Source Skeleton

### `src/index.ts`
Assigns the GAS entry-point functions to `globalThis` so they are accessible by the Apps Script runtime after the IIFE executes. `globalThis` must be cast to `any` because strict TypeScript does not allow arbitrary property assignment on `globalThis` without a declaration:
```typescript
import { setup, update, populateConfig } from './update';

const g = globalThis as any;
g.setup = setup;
g.update = update;
g.populateConfig = populateConfig;
```

### `src/update.ts`
Stub exports for the three GAS functions — to be filled with migrated logic:
```typescript
export function setup(): void {}
export function update(): void {}
export function populateConfig(): void {}
```

### `src/api.ts`
Placeholder for BitJita API helpers:
```typescript
export function fetchJson(url: string): unknown {
  return JSON.parse(UrlFetchApp.fetch(url).getContentText());
}
```

### `src/sheet.ts`
Placeholder for spreadsheet helpers:
```typescript
export function getSheet(name: string): GoogleAppsScript.Spreadsheet.Sheet {
  return SpreadsheetApp.getActiveSpreadsheet().getSheetByName(name)!;
}
```

### `dist/appsscript.json`
GAS manifest — committed to the repo, not generated:
```json
{
  "timeZone": "Europe/Helsinki",
  "exceptionLogging": "STACKDRIVER",
  "runtimeVersion": "V8"
}
```

## Migration Notes

- The existing `Import.gs` remains in the repo root until an explicit migration task is undertaken
- The `test()` function in `Import.gs` is intentionally excluded from the skeleton — it is empty and not needed
- When migrating, the logic in `Import.gs` maps to: API calls → `api.ts`, sheet operations → `sheet.ts`, orchestration → `update.ts`

## Local Setup Prerequisites

Before using clasp, a developer must authenticate once:

```sh
npx clasp login
```

This creates `~/.clasprc.json` with OAuth credentials. The `scriptId` for `.clasp.json` can be found in the Apps Script editor under **Project Settings** (the gear icon), or from the URL: `https://script.google.com/home/projects/<scriptId>/edit`.

## Handling the Existing Remote `Import.gs`

After the first `clasp push`, both `Import.gs` (existing remote file) and `Code.js` (newly pushed) will coexist in the Apps Script project, causing duplicate function definitions for `setup`, `update`, and `populateConfig`. Before or immediately after the first push, the remote `Import.gs` must be deleted manually via the Apps Script editor.

GAS OAuth scopes (`UrlFetchApp`, `SpreadsheetApp`) are inferred automatically from the code — no explicit `oauthScopes` field is needed in `appsscript.json` for local development.

## Future: Automated Deployments

The `deploy` script is the hook for CI. Clasp authenticates via OAuth 2.0 credentials stored in `~/.clasprc.json`. For CI (e.g., GitHub Actions), the approach is to export the contents of `~/.clasprc.json` as an encrypted secret, restore it to the runner's home directory before running `clasp push`, and then run `npm run deploy`. Note: service accounts cannot own Apps Script projects and cannot be used with clasp — OAuth credentials are required. Known limitation: the OAuth tokens in `.clasprc.json` expire and must be refreshed periodically; re-running `clasp login` locally and updating the CI secret is required when this happens.
